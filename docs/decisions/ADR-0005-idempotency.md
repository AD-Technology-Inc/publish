# ADR-0005: Atomic Redis Idempotency Key Strategy

## Context
Under at-least-once delivery, multiple workers may attempt to process duplicate job deliveries generated by client network retries or stream re-claims.

## Problem
Duplicate execution of publishing operations triggers duplicate side-effects on social networks (e.g., publishing identical posts twice to Facebook or LinkedIn).

## Decision
Enforce atomic idempotency key assertions at worker entry using `IdempotencyMiddleware` (`SET idempotency:{key} 1 NX EX 86400`).

## Alternatives Considered
- **Database Unique Constraints**: Relying solely on SQL `UNIQUE` constraints. (Rejected because non-database side-effects like social network API calls cannot be rolled back by SQL database engines).
- **Distributed Lock Manager (Redlock)**: Complex multi-node locking. (Rejected as single Redis atomic `SET NX` provides sufficient isolation for single-instance Redis deployments).

## Consequences
- Deduplication occurs in sub-milliseconds before executing any external side-effects.
- Idempotency keys auto-expire after 24 hours (`ex=86400`) to bound Redis RAM footprint.

## Tradeoffs
- Requires clients to generate unique idempotency keys or rely on system-generated UUIDs.
